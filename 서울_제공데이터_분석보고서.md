## 1. 분석 범위

- 선정 권역: 서울
- 데이터 출처: 한국관광공사 TourAPI 4.0
- 실제 업로드된 JSON: 7개
- 실제 로드된 항목: 6,518건
- 업로드 파일: 관광지, 레포츠, 문화시설, 쇼핑, 숙박, 여행코스, 축제공연행사

## 2. 우선 확인된 불일치

- SOURCE.md에는 서울 데이터가 8개 파일, 총 8,150건으로 기재되어 있습니다.
- 현재 업로드된 파일은 7개, 총 6,518건입니다.
- `서울_음식점.json` 1,632건이 현재 업로드 목록에서 빠져 있습니다.
- 나머지 7개 파일은 각 파일의 `total` 값과 실제 `items` 배열 길이가 모두 일치합니다.

## 4. 개발 시 중요한 데이터 특성

### 공통

- `mapx`, `mapy`는 문자열이므로 지도 사용 전에 숫자로 변환해야 합니다.
- `contentTypeId`는 파일 최상위에서 일부 숫자형으로 저장되어 있고, 항목 내부 `contenttypeid`는 문자열이므로 비교 시 타입을 통일해야 합니다.
- `cat1~3`는 다수 항목에서 비어 있으나 `lclsSystm1~3`는 모든 업로드 파일에서 채워져 있습니다. 분류·필터는 `lclsSystm` 계열을 우선 사용하는 편이 안전합니다.
- 대표 이미지가 없는 항목에는 기본 대체 이미지를 표시해야 합니다.
- 일부 대표 이미지 URL은 `http://`이므로 HTTPS로 배포되는 Netlify에서 혼합 콘텐츠 문제가 생길 수 있습니다.
- 업로드된 7개 파일 사이에서 중복 `contentid`는 발견되지 않았습니다.

### 유형별

- 쇼핑 데이터가 4,368건으로 전체 업로드 데이터의 대부분을 차지하므로 전체 데이터를 한 번에 화면에 렌더링하지 말고 필터·페이지네이션·지연 로딩을 적용하는 것이 좋습니다.
- 숙박은 대표 이미지가 없는 항목이 231건으로 많아 카드 UI의 대체 이미지 처리가 필수입니다.
- 여행코스 51건은 `addr1`과 `zipcode`가 전부 비어 있으므로 주소 기반 구 필터를 적용할 수 없습니다. 좌표 또는 법정동 코드로 별도 처리해야 합니다.
- 관광지·레포츠·문화시설·숙박의 `tel`은 전부 비어 있습니다. 전화번호는 필수 표시 항목으로 설계하지 않는 편이 좋습니다.
- 축제공연행사 데이터에는 전화번호가 모두 존재하지만, 현재 제공 스키마에는 행사 시작일·종료일 필드가 없습니다. 따라서 현재 파일만으로는 정확한 축제 캘린더 구현이 어렵습니다.

## 5. MVP에서 권장하는 사용 범위

1. 메인 추천/카드: 관광지, 문화시설, 레포츠, 여행코스
2. 지도 핀: 좌표가 유효한 관광지·문화시설·레포츠·숙박·쇼핑·축제
3. 유형 필터: `contentType` 또는 `contentTypeId`
4. 세부분류 필터: `lclsSystm1~3`
5. 구별 필터: 주소에서 자치구를 추출하되 여행코스는 제외하거나 별도 분류
6. 챗봇 컨텍스트: 전체 6천여 건을 그대로 프롬프트에 넣지 않고 검색·필터 후 상위 일부 항목만 전달
